## Overview

M02 is the Question Bank â€” a persistent, deduplicated store of every question ever generated by Claude AI for Learnfyra. It provides a pluggable two-adapter architecture (local JSON files for development, DynamoDB for AWS) controlled by the `QB_ADAPTER` environment variable. M03 (Worksheet Generator) consults M02 before calling the Claude AI API, using banked questions to fill as much of a generation request as possible and only calling the AI for the remaining gap. M02 is a pure backend module with no user-facing UI.

## User Stories

### US-QB-001: Reuse Previously Generated Questions
**As a** teacher
**I want to** receive worksheets quickly for commonly requested topics
**So that** generation latency and Claude API cost are reduced by reusing banked questions instead of regenerating them
**Priority:** P1

### US-QB-002: Prevent Duplicate Questions in the Bank
**As a** developer
**I want to** ensure the same question text is never stored twice for the same grade/subject/topic/type combination
**So that** the bank stays clean and question variety metrics are accurate
**Priority:** P0

### US-QB-003: Full Parity Between Local and AWS Adapters
**As a** developer running the app locally
**I want to** use a JSON file adapter that satisfies the same interface as the DynamoDB adapter
**So that** I can develop and test the full bank-first pipeline without any AWS credentials or network access
**Priority:** P0

### US-QB-004: Track Question Usage for Analytics
**As a** platform administrator
**I want to** see how many times each banked question has been included in a generated worksheet
**So that** I can identify frequently used topics and plan curriculum improvements
**Priority:** P2

## Functional Requirements

### REQ-QB-001: Adapter Factory
**Priority:** P0
**Tasks:** QB-IMPL-004

`src/questionBank/index.js` SHALL export a factory function that reads the `QB_ADAPTER` environment variable and returns the correct adapter singleton. Valid values are `local` (the default when the variable is absent) and `dynamodb`. If `QB_ADAPTER` is set to any other value, the factory SHALL throw an error at initialization time (not at call time) so that misconfiguration is caught during Lambda cold start.

Both adapters SHALL implement the following three-method interface exactly. No method may be missing or have a different signature:

```
getQuestions({ grade, subject, topic, type, difficulty, count }) â†’ Promise<Question[]>
saveQuestion(question)                                           â†’ Promise<{ questionId: string, dedupeHash: string }>
questionExists(dedupeHash)                                       â†’ Promise<boolean>
```

### REQ-QB-002: Local Adapter (QB_ADAPTER=local)
**Priority:** P0
**Tasks:** QB-IMPL-002

The local adapter in `src/questionBank/localAdapter.js` SHALL read and write question arrays as JSON files in `worksheets-local/question-bank/`. Filename convention: `grade{N}-{subject-lowercase-hyphenated}-{topic-lowercase-hyphenated}.json` (example: `grade3-math-multiplication.json`). Each file contains a JSON array of question objects conforming to REQ-QB-005.

Behavior per method:
- `getQuestions`: Load the file, filter in memory by `type` and `difficulty`, and return up to `count` items. If fewer than `count` items are available, return all available without error. If the file does not exist, return an empty array.
- `saveQuestion`: Read the existing file (or start with `[]`), compute the dedupeHash via REQ-QB-004, check for an existing match, and append only if no match exists. Write the updated array back. Return the `questionId` and `dedupeHash`.
- `questionExists`: Read the file, compute SHA256 of the incoming parameters, and return `true` if any stored item's `dedupeHash` matches.

The local adapter SHALL operate without any AWS credentials or network access.

### REQ-QB-003: DynamoDB Adapter (QB_ADAPTER=dynamodb)
**Priority:** P0
**Tasks:** QB-IMPL-003, QB-CDK-001, QB-CDK-002

The DynamoDB adapter in `src/questionBank/dynamoAdapter.js` SHALL use `@aws-sdk/client-dynamodb` and `@aws-sdk/lib-dynamodb` with lazy import for cold-start optimization. The table name SHALL be read from the `QUESTION_BANK_TABLE_NAME` environment variable (injected by CDK). It SHALL never be hardcoded.

Behavior per method:
- `getQuestions`: Query GSI-1 with `lookupKey = "grade#{grade}#subject#{subject}#topic#{topic}"` and `KeyConditionExpression typeDifficulty BEGINS_WITH "{type}#{difficulty}"`. Apply `Limit = count`. Return items as plain JavaScript objects.
- `saveQuestion`: Call `questionExists(dedupeHash)` first. If already present, return the existing `questionId` without writing. If new, generate a UUID v4 `questionId`, set `timesUsed = 0`, and execute a DynamoDB `PutItem`.
- `questionExists`: Query GSI-1 with the computed `lookupKey` and filter on `dedupeHash`. Return `true` if any item is found.

In addition, the DynamoDB adapter SHALL expose a fourth method `incrementUsage(questionId)` that issues a non-blocking `UpdateItem` to atomically increment `timesUsed` by 1. This method is called by the M03 generator after worksheet assembly and SHALL NOT be awaited on the critical generation path (fire-and-forget pattern).

### REQ-QB-004: dedupeHash Computation
**Priority:** P0
**Tasks:** QB-IMPL-005

The canonical dedupeHash function SHALL live in `src/questionBank/utils.js` and SHALL be the only implementation used by both adapters. The algorithm:

```
normalized = [
  question.grade,
  question.subject.toLowerCase(),
  question.topic.toLowerCase(),
  question.type,
  question.question.toLowerCase().replace(/\s+/g, ' ').trim()
].join('|')

dedupeHash = hex digest of SHA256(normalized)
```

Use the `createHash` function from the Node.js built-in `crypto` module. No external hashing dependency is permitted. The function SHALL be exported as `computeDedupeHash(question)` and SHALL accept any object with `grade`, `subject`, `topic`, `type`, and `question` fields.

### REQ-QB-005: Stored Question Schema
**Priority:** P0
**Tasks:** QB-IMPL-002, QB-IMPL-003

Every question stored in the bank SHALL conform to this schema. All fields are required unless noted as optional.

| Field | Type | Required | Constraint |
|---|---|---|---|
| questionId | string | Yes | UUID v4 |
| dedupeHash | string | Yes | SHA256 hex, exactly 64 characters |
| grade | number | Yes | Integer 1â€“10 |
| subject | string | Yes | Enum: Math / ELA / Science / Social Studies / Health |
| topic | string | Yes | From CURRICULUM map in `src/ai/topics.js` |
| type | string | Yes | Enum: multiple-choice / fill-in-the-blank / short-answer / true-false / matching / show-your-work / word-problem |
| difficulty | string | Yes | Enum: Easy / Medium / Hard |
| question | string | Yes | Question text |
| options | array | Multiple-choice only | Exactly 4 strings: ["A. ...", "B. ...", "C. ...", "D. ..."] |
| answer | string | Yes | Correct answer string |
| explanation | string | Yes | Answer key explanation |
| points | number | Yes | Default: 1 |
| standards | array of strings | Yes | CCSS / NGSS / NHES / C3 / NHES codes |
| createdAt | string | Yes | ISO-8601 timestamp |
| generatedByModel | string | Yes | Claude model ID (e.g., `claude-sonnet-4-20250514`) |
| timesUsed | number | Yes | Starts at 0 on save; incremented via `incrementUsage` |

The `options` field SHALL NOT be present on any question type other than `multiple-choice`.

### REQ-QB-006: Bank-First Worksheet Assembly Integration
**Priority:** P0
**Tasks:** M03-BE-05, M03-BE-06 (both BLOCKED on QB-IMPL-001 through QB-IMPL-004 completion)

The worksheet generator `src/ai/generator.js` SHALL be updated to integrate the question bank as Steps 2â€“7 of the generation pipeline documented in `02-modules/worksheet-generator.md`. The integration points are:

**Step 2 â€” Question type breakdown:** For each question type applicable to the requested grade/subject (from `getQuestionTypesForGradeSubject` in `src/ai/topics.js`), compute `targetCount[type] = Math.ceil(questionCount * typeRatio[type])`.

**Step 3 â€” Bank query per type:** Call `QB.getQuestions({ grade, subject, topic, type, difficulty, count: targetCount[type] })` for each type. Collect all returned questions.

**Step 4 â€” generationMode decision:**
- `totalBanked >= questionCount` â†’ `generationMode = 'bank-only'` (skip Claude AI)
- `totalBanked === 0` â†’ `generationMode = 'ai-only'` (call Claude AI for all questions)
- `0 < totalBanked < questionCount` â†’ `generationMode = 'mixed'` (call Claude AI for `gap = questionCount - totalBanked` questions)

**Step 7 â€” Save new AI questions:** For each AI-generated question, compute `dedupeHash`, call `QB.questionExists(dedupeHash)`, and call `QB.saveQuestion(question)` only if the result is `false`.

**provenanceLevel** SHALL be set:
- All questions from bank â†’ `full-bank`
- Mix of bank and AI â†’ `partial-bank`
- All from AI â†’ `full-ai`

Both `generationMode` and `provenanceLevel` SHALL be written to the generate response body, to `metadata.json`, and to `solve-data.json`.

### REQ-QB-007: Environment Variable Requirements
**Priority:** P0
**Tasks:** QB-CDK-003

The following environment variables SHALL be defined:
- `QB_ADAPTER`: value `local` or `dynamodb`. When absent, defaults to `local`.
- `QUESTION_BANK_TABLE_NAME`: DynamoDB table name injected by CDK for AWS deployments. Must be absent (or ignored) when `QB_ADAPTER=local`.

The factory SHALL NOT throw if `QUESTION_BANK_TABLE_NAME` is absent when `QB_ADAPTER=local`.

## Local Development Requirements

This section applies before any AWS work begins. All acceptance criteria above MUST pass on `http://localhost:3000` with `QB_ADAPTER=local` before any DynamoDB or CDK work begins.

### Local Runtime Control
- `QB_ADAPTER=local` (set in `.env`) activates the JSON file adapter. This is the default when the variable is absent.
- `QB_ADAPTER=dynamodb` activates the DynamoDB adapter. When `NODE_ENV=development`, the DynamoDB adapter SHALL use `DB_ENDPOINT=http://localhost:8000` (dynamodb-local Docker container) instead of real AWS DynamoDB.
- `APP_RUNTIME=local` (set in `.env`) is the master switch. When set, the application expects no AWS credentials.

### Environment Variables for Local Development
```
QB_ADAPTER=local                         # Use JSON file adapter (no Docker needed)
# or
QB_ADAPTER=dynamodb                      # Use DynamoDB adapter pointed at Docker
DB_ENDPOINT=http://localhost:8000        # dynamodb-local endpoint (DynamoDB adapter only)
NODE_ENV=development                     # Causes DynamoDB adapter to prefer DB_ENDPOINT
QUESTION_BANK_TABLE_NAME=LearnfyraQuestionBank-local  # Table name in dynamodb-local
```

### Local JSON File Adapter â€” No Setup Required
When `QB_ADAPTER=local`, no Docker container, no DynamoDB table, and no AWS credentials are needed. The adapter reads and writes files under `worksheets-local/question-bank/`. This directory is created on first write if it does not exist.

Before running tests with the local adapter, seed the question bank with sample data:
```
node scripts/seed-question-bank.js
```
This populates `worksheets-local/question-bank/` with enough sample questions to exercise the bank-first pipeline (see SETUP-004 in the master task list).

### DynamoDB Local â€” Required Only for QB_ADAPTER=dynamodb
If testing the DynamoDB adapter locally, `dynamodb-local` must be running before any test that exercises `QB_ADAPTER=dynamodb`. Start it with:
```
docker run -p 8000:8000 amazon/dynamodb-local
```

Create the question bank table in dynamodb-local before running tests:
```
aws dynamodb create-table \
  --table-name LearnfyraQuestionBank-local \
  --attribute-definitions \
    AttributeName=questionId,AttributeType=S \
    AttributeName=lookupKey,AttributeType=S \
    AttributeName=typeDifficulty,AttributeType=S \
  --key-schema AttributeName=questionId,KeyType=HASH \
  --global-secondary-indexes '[{
    "IndexName": "GSI-1",
    "KeySchema": [
      {"AttributeName": "lookupKey", "KeyType": "HASH"},
      {"AttributeName": "typeDifficulty", "KeyType": "RANGE"}
    ],
    "Projection": {"ProjectionType": "ALL"}
  }]' \
  --billing-mode PAY_PER_REQUEST \
  --endpoint-url http://localhost:8000
```

This table creation is also encapsulated in `scripts/bootstrap-local-db.js` (SETUP-002).

### Local Test Sequence
Before any AWS work begins, verify the following sequence on `http://localhost:3000`:
1. Start the server with `QB_ADAPTER=local` in `.env`.
2. Call `POST /api/generate` for any grade/subject/topic â€” confirm `generationMode: "ai-only"` and new files appear under `worksheets-local/question-bank/`.
3. Call `POST /api/generate` again with identical parameters â€” confirm at least some questions are served from the bank (`generationMode: "mixed"` or `"bank-only"` depending on question count).
4. Run `npm test` â€” all QB unit tests pass with `QB_ADAPTER=local`.

## Out of Scope
- Per-student question repeat caps â€” `timesUsed` is tracked but not acted upon in Phase 1 (deferred to Phase 2).
- Manual curation or quality-scoring UI for banked questions.
- Bulk import or export of question bank data.
- Cross-topic or cross-grade question reuse; a question is only returned for queries matching its exact grade/subject/topic combination.
- Question versioning or edit workflow â€” questions are write-once in Phase 1.
- DynamoDB Streams or event-driven workflows triggered on question bank writes.
- A random-sample query strategy for popular topics â€” questions are returned in DynamoDB query order in Phase 1.

## Open Questions

1. The DynamoDB design in `03-data-design/dynamodb-design.md` does not include a dedicated GSI on `dedupeHash`. The `questionExists` method as described performs a Query on GSI-1 followed by a client-side filter on `dedupeHash`, which is O(results). Should a dedicated `dedupeHash-index` GSI be added to make `questionExists` an O(1) GetItem? Decision needed before QB-CDK-001 is deployed.
2. The `typeRatio` map governing how many questions of each type to request per generation must be defined in `src/ai/topics.js`. This is a DBA responsibility. Until it is specified, M03-BE-05 cannot compute `targetCount[type]`. Who owns this and when will it be available?
3. Should `getQuestions` return a deterministic first-N result (DynamoDB query order) or a random sample from all available questions for that topic? Random sampling would improve question variety for repeated generation requests but requires a more complex DynamoDB access pattern (e.g., a random offset or Scan with filter). Decision needed before QB-IMPL-003.
4. The `incrementUsage` call on banked questions â€” should it be fire-and-forget (non-blocking) or awaited? The recommendation in this spec is fire-and-forget to avoid adding latency to the generation response path, but this needs explicit team confirmation.
