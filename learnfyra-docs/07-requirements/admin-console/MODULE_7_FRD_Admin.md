## 1. Document Header

This document is the authoritative Functional Requirements Document for Module 7: Admin Console. It covers both Phase 2A (Platform Admin — super_admin tier) and Phase 2B (School Admin — school_admin tier) of the Learnfyra admin capability. It supersedes the earlier stub requirements in `07-requirements/admin/README.md`, which was a preliminary outline and not a design-resolved FRD.

---

## 2. Executive Summary

Module 7 introduces administrative governance tooling for the Learnfyra platform. It is intentionally scoped to Phase 2 because it depends on all Modules 1 through 6 being operational: admin workflows presuppose an established user base, a functioning worksheet generator, a working solve-and-submit flow, progress tracking data, teacher and parent capabilities, and a stable infrastructure baseline. Building admin tooling before the platform itself has live users would produce premature controls with no data to operate on.

The module delivers a two-tier admin hierarchy. The first tier — super_admin — is Learnfyra's internal operations team, numbering between two and five globally. These individuals exercise maximum trust over the entire platform: they manage users, moderate AI-generated content in the Question Bank, monitor and control AI generation costs, edit system configuration, manage the school registry, and handle COPPA deletion requests that carry legal weight. The second tier — school_admin — is customer-facing. A school admin manages teachers and students within exactly one school. They have no visibility into platform-wide data, cannot touch configuration, and cannot perform user deletions. Their operational concerns are teacher onboarding, class-level oversight, school-wide analytics, and worksheet distribution. The two tiers are separate roles stored on the User record and must never be conflated: a school admin is not a teacher, and a teacher is not a school admin. This separation keeps data scope, trust level, and operational surface clearly defined, which is essential for a platform that handles data belonging to children under the age of thirteen.

## 3. Scope

### 3.1 In-Scope Phase 2A — Platform Admin

- super_admin role definition on the User record
- CDK bootstrap flow that seeds the first super_admin at deployment time
- Separate admin console Vite application deployed to its own CloudFront distribution, IP-restricted for super_admin access
- User management: search, view full profile, suspend, unsuspend, force immediate logout via Cognito Admin API, role change, COPPA deletion
- Question Bank moderation: flag, unflag, soft-delete (status field amendment to QuestionBank table)
- AI cost dashboard: read-only, query-time aggregation from GenerationLog table, daily token budget ceiling enforcement
- Config table editor with type validation and model allowlist enforcement
- School creation and school_admin assignment
- AuditLog table: append-only, covers all super_admin state-changing actions
- ComplianceLog table: write-before-delete ordering for COPPA deletion, permanent retention
- Daily token budget ceiling enforcement: amendment to Module 2 worksheet generation Lambda
- Lambda Authorizer amendment: suspended flag check on every authenticated request

### 3.2 In-Scope Phase 2B — School Admin

- school_admin role definition and assignment by super_admin
- Teacher invite flow: school-level invite code generated by school_admin or super_admin
- Teacher roster management: view active teachers and their classes, remove teacher from school
- School-wide student roster: read-only view across all classes in the school
- School-level analytics dashboard: subject accuracy, grade-level trends, teacher completion rates
- Bulk worksheet assignment across multiple classes within the school
- School configuration: grade range and active subjects

### 3.3 Out-of-Scope Phase 3 (with reasoning)

| Item | Reason Deferred |
|---|---|
| District admin tier | No paying district customers at Phase 2 launch; adds a third role layer with significant data-scoping complexity. Deferred until a district contract makes it necessary. |
| Admin user impersonation (act-as for debugging) | High security risk if not implemented carefully. Requires audit trail for impersonated actions, session separation, and legal review. Deferred until a specific debugging use case justifies the risk. |
| Automated AI-driven content quality scoring for QuestionBank entries | Would require a separate AI model call per question at ingest time, adding cost and latency. Manual moderation (flag/unflag) is sufficient for Phase 2 volume. |
| Cost alert webhooks (Slack/email on budget threshold breach) | Infrastructure for outbound webhook delivery is not yet in the stack. Phase 2A provides the dashboard and 503 enforcement; proactive alerting deferred to Phase 3. |
| Multi-school teacher accounts | The SchoolUserLink model supports it at the data layer, but the UI flows and permission logic for a teacher appearing in two schools add significant complexity. Deferred until a use case arises. |
| Pre-aggregated DailyCostSummary table | Query-time computation from GenerationLog GSIs is acceptable for Phase 2A volume. If 30-day window queries exceed 2 seconds in production, Phase 3 will introduce this table. |
| Admin mobile interface | Admin console is desktop-only in Phase 2. Super admins are operations staff working on workstations. Mobile admin creates additional attack surface with limited benefit. |
| CSV/PDF export of analytics or compliance data | File export requires presigned URL generation, S3 temporary storage, and cleanup. Deferred until customer demand is confirmed. |

### 3.4 Architectural Constraints

The admin console is a separate Vite application deployed as a separate CloudFront distribution. It shares the same API Gateway and Lambda functions as the main platform but is deployable independently via its own CDK stack. This separation enables IP restriction at the CloudFront level for super_admin access without affecting the main platform's availability or routing.

The AuditLog must be enforced as append-only at the application layer. No Lambda function, IAM role, or CDK construct may expose a delete or update operation on the AuditLog table. This is not achievable via IAM alone on DynamoDB without impacting batch operations, so the constraint is a hard application-layer rule: no handler anywhere in the codebase may call DeleteItem or UpdateItem against the AuditLog table.

The ComplianceLog must be permanently retained. There is no TTL on this table. A super_admin may read but never delete ComplianceLog records through the admin console. Manual DynamoDB deletion by a developer is a policy violation that must be documented in the operations runbook.

## 4. User Roles and Personas

### super_admin

A super_admin is a member of Learnfyra's internal operations team. There will be between two and five of these globally at any time. The first super_admin is provisioned automatically by the CDK deployment stack using a Cognito user email passed as a CDK parameter — no manual DynamoDB write or Cognito console action is required. All subsequent super_admins are assigned by an existing super_admin through the admin console. This role is never self-service: no user can request it, and no UI flow exists by which a user could acquire it without an existing super_admin explicitly granting it. A super_admin has maximum trust over the platform. Their primary operational concerns are: protecting children's data (COPPA deletion), ensuring content quality does not degrade (Question Bank moderation), keeping AI generation costs within budget (cost dashboard and token budget ceiling), maintaining system configuration (Config table editor), onboarding school customers (school creation and school_admin assignment), and preserving an immutable audit trail of all administrative actions.

### school_admin

A school_admin is a customer-facing role created exclusively by a super_admin. A school admin is always affiliated with exactly one school at a time. They cannot create their own school, cannot self-register as a school admin, and cannot see or affect any users outside their school. Their primary operational concerns are: onboarding teachers to their school, monitoring teacher and student activity, running school-wide analytics to identify weak subject-grade combinations, and distributing worksheet assignments across multiple classes efficiently. A school admin exercises the trust of a school operator — sufficient to manage their own institution's roster and settings, but with zero cross-institution or platform-wide access.

### 5.1 Role Bootstrap and Assignment

**FR-A-001** | CDK bootstrap seeds the first super_admin at deploy time | Must |
- Given a valid Cognito user email is supplied as a CDK parameter at stack deploy time, when the CDK stack is successfully deployed to any environment, then a DynamoDB User record for that email is written with role=super_admin before any other CDK resource creation completes.
- Given the CDK bootstrap parameter is absent or contains an email not matching any Cognito user, when the stack is deployed, then the deployment fails with a descriptive error and no partial super_admin record is written.
- Given a deployed environment, when a developer queries the Users table for any record with role=super_admin, then exactly one such record exists immediately after first deployment.

**FR-A-002** | super_admin can assign the super_admin role to an existing user | Must |
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/role with role=super_admin, when the target user exists and is not already a super_admin, then the User record is updated with role=super_admin, and an AuditLog record is written with action=ROLE_CHANGE, beforeState containing the previous role, and afterState containing super_admin.
- Given an authenticated super_admin attempts to call PATCH /admin/users/{userId}/role, when the caller's JWT does not contain role=super_admin, then the Lambda Authorizer returns 403 before the handler executes.
- Given a super_admin assigns a new super_admin role, when the request succeeds, then an AuditLog record is written containing actorId, actorRole=super_admin, action=ROLE_CHANGE, targetEntityId=userId, beforeState, and afterState.

**FR-A-003** | No user may self-assign an admin role | Must |
- Given any authenticated user calls PATCH /admin/users/{userId}/role, when {userId} matches the caller's own Cognito sub, then the request is rejected with 403 regardless of the caller's role.

**FR-A-004** | super_admin can assign or revoke the school_admin role | Must |
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/role with role=school_admin, when the target user exists, then the User record is updated with role=school_admin, and an AuditLog record is written with action=ROLE_CHANGE.
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/role with role=student or role=teacher (downgrade), when the target user currently holds role=school_admin, then the User record is updated to the new role, any SchoolUserLink records for this user are set to status=removed, and an AuditLog record is written.

### 5.2 User Management

**FR-A-010** | super_admin can search and list users | Must |
- Given an authenticated super_admin calls GET /admin/users with optional query parameters search, role, and suspended, when the request is processed, then the response contains a paginated list of User records matching all supplied filters, returning at most 50 records per page.
- Given the search parameter is provided, when the query is processed, then the results include only users whose email, display name, or Cognito sub contains the search string (case-insensitive).
- Given the suspended=true filter is provided, when the query is processed, then only users with suspended=true on their User record are returned.
- Given no filters are provided, when the query is processed, then all users are returned in paginated form ordered by createdAt descending.
- Given a caller with role other than super_admin calls GET /admin/users, then the Lambda Authorizer returns 403.

**FR-A-011** | super_admin can view a user's full profile | Must |
- Given an authenticated super_admin calls GET /admin/users/{userId}, when the User record exists, then the response includes all profile fields: userId, email, displayName, role, suspended, createdAt, lastLoginAt, and a summary of activity (worksheet generation count from GenerationLog, solve attempt count from WorksheetAttempt).
- Given a caller with role=school_admin calls GET /admin/users/{userId} for a user not in their school, then the Lambda Authorizer returns 403.
- Given the userId does not exist in DynamoDB, when the request is processed, then the response is 404.

**FR-A-012** | super_admin can suspend a user | Must |
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/suspend, when the User record exists and suspended is currently false, then the User record is updated with suspended=true, and an AuditLog record is written with action=USER_SUSPENDED, afterState containing suspended=true.
- Given the target user is already suspended, when the suspend request is received, then the response is 409 Conflict.
- Given a user is suspended, when they make any subsequent authenticated request and the Lambda Authorizer's in-memory cache has expired (after at most 5 minutes), then the Authorizer returns 401 for that request.
- Given a suspend action succeeds, then an AuditLog record is written with actorId, actorRole=super_admin, action=USER_SUSPENDED, targetEntityId=userId, ipAddress, userAgent, and timestamp.

**FR-A-013** | Lambda Authorizer checks suspended flag on every authenticated request | Must |
- Given the Lambda Authorizer receives a valid JWT for a user whose User record has suspended=true, when the cached suspension result for that userId has expired or is absent, then the Authorizer performs a DynamoDB GetItem on the Users table to retrieve the suspended field.
- Given the retrieved suspended field is true, when the Authorizer evaluates the policy, then it returns a Deny policy and the caller receives 401.
- Given the retrieved suspended field is false, when the Authorizer evaluates the policy, then the suspension check result is cached in Lambda in-memory for 5 minutes and the request proceeds.
- Given the Authorizer has a cached suspension result for a userId, when a new request arrives for that userId within 5 minutes of the last check, then the Authorizer uses the cached result without performing a new DynamoDB read.

**FR-A-014** | super_admin can unsuspend a user | Must |
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/unsuspend, when the User record exists and suspended is currently true, then the User record is updated with suspended=false, and an AuditLog record is written with action=USER_UNSUSPENDED.
- Given the target user is not currently suspended, when the unsuspend request is received, then the response is 409 Conflict.
- Given an unsuspend action succeeds, then an AuditLog record is written with actorId, actorRole=super_admin, action=USER_UNSUSPENDED, targetEntityId=userId, ipAddress, userAgent, and timestamp.

**FR-A-015** | super_admin can force immediate logout via Cognito | Must |
- Given an authenticated super_admin calls POST /admin/users/{userId}/force-logout, when the request is received, then the system displays a confirmation dialog before the API call is made.
- Given the super_admin confirms the force-logout action, when the Lambda handler executes, then it calls the Cognito AdminUserGlobalSignOut API for the target user's Cognito username, invalidating all active refresh tokens.
- Given the Cognito call succeeds, when the handler completes, then an AuditLog record is written with action=FORCE_LOGOUT, targetEntityId=userId, actorId, actorRole=super_admin, ipAddress, userAgent, and timestamp.
- Given the Cognito call fails, when the handler catches the error, then the error is returned as 502 with a human-readable message and no AuditLog record is written (because the action did not succeed).
- Given a caller with any role other than super_admin calls POST /admin/users/{userId}/force-logout, then the Lambda Authorizer returns 403.

**FR-A-016** | super_admin can change a user's role | Must |
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/role with a valid role value (student, teacher, parent, school_admin, super_admin), when the target user exists, then the User record is updated with the new role, and an AuditLog record is written with action=ROLE_CHANGE, beforeState containing the previous role, and afterState containing the new role.
- Given an invalid role value is supplied, when the request is processed, then the response is 400 with a clear validation message listing the allowed values.
- Given the role change succeeds, then an AuditLog record is written with actorId, actorRole=super_admin, action=ROLE_CHANGE, targetEntityId=userId, beforeState, afterState, ipAddress, userAgent, and timestamp.

### 5.3 COPPA Deletion Flow

**FR-A-020** | super_admin can initiate a COPPA deletion request | Must |
- Given an authenticated super_admin calls DELETE /admin/users/{userId}, when the request body contains a valid confirmationToken (a token issued by a prior confirmation step), then the deletion flow begins.
- Given the target user does not exist, when the deletion is requested, then the response is 404 and no deletion is performed.
- Given a caller with any role other than super_admin calls DELETE /admin/users/{userId}, then the Lambda Authorizer returns 403.

**FR-A-021** | COPPA deletion UI requires double-confirmation | Must |
- Given a super_admin initiates a COPPA deletion from the admin console UI, when the deletion is triggered, then the UI presents a modal requiring the super_admin to type the target user's exact email address before a confirmation token is issued.
- Given the typed email does not exactly match the target user's email, when the super_admin attempts to confirm, then the confirmation token is not issued and the deletion does not proceed.
- Given the email matches exactly, when the super_admin confirms, then a short-lived confirmation token is issued client-side and passed in the DELETE request body.

**FR-A-022** | ComplianceLog record is written before deletion begins | Must |
- Given a COPPA deletion is initiated with a valid confirmation token, when the handler begins execution, then a ComplianceLog record is written to DynamoDB with requestType=coppa-deletion, requestedBy (super_admin userId), targetUserId, startedAt, legalBasis, and a status of in-progress, before any deletion step begins.
- Given the ComplianceLog write fails for any reason, when the error is caught, then no deletion steps are executed and the handler returns 500 with an error message indicating the compliance record could not be written.
- Given the ComplianceLog write succeeds, then and only then do the deletion steps proceed in the specified order.

**FR-A-023** | COPPA deletion executes in defined order | Must |
- Given a valid ComplianceLog record has been written, when the deletion handler proceeds, then it executes deletion steps in this exact order: (1) delete all WorksheetAttempt records for the targetUserId, (2) delete all UserProgress records for the targetUserId, (3) delete all ParentChildLink records where the user is either parent or child, (4) delete all StudentAssignmentStatus records for the targetUserId, (5) remove the user from all Class rosters by clearing any SchoolUserLink records with status=removed, (6) delete the User record from DynamoDB, (7) delete the Cognito user account.
- Given any deletion step fails, when the error is caught, then the error details (step name, error message, partial count) are appended to the ComplianceLog record's errorState field, execution halts for that step, subsequent steps do not execute, and the handler returns 500.
- Given all deletion steps succeed, when the handler completes, then the ComplianceLog record is updated with completedAt, deletedEntities listing each entity type with its count, and status=completed.
- Given the deletion completes (success or partial failure), then an AuditLog record is written with action=COPPA_DELETION, targetEntityId=userId, actorId, actorRole=super_admin, ipAddress, userAgent, and timestamp.

**FR-A-024** | Partial COPPA deletions are recorded and do not auto-retry | Must |
- Given a COPPA deletion step fails midway, when the handler has recorded the error in the ComplianceLog, then no automatic retry is attempted and the super_admin is presented with the error state in the admin console.
- Given a partial deletion is recorded in the ComplianceLog, when a super_admin queries GET /admin/compliance-log, then the partial deletion record is visible with its errorState field populated.

### 5.4 Question Bank Moderation

**FR-A-030** | QuestionBank table gains a status field | Must |
- Given the Module 7 data model migration runs, when a new Question record is written to the QuestionBank table, then it includes a status field with default value=active.
- Given existing Question records were written before Module 7 migration, when they are read by the Module 2 Step Functions bank lookup query, then only records with status=active are returned (the GSI query filter must include a condition on status=active).
- Given the status field amendment is deployed, when the Module 2 Step Functions workflow executes a bank lookup via GSI, then the query explicitly filters on status=active and excludes any record with status=flagged or status=deleted.

**FR-A-031** | super_admin can flag a QuestionBank entry | Must |
- Given an authenticated super_admin calls PATCH /admin/question-bank/{questionId}/flag, when the Question record exists and has status=active, then the status field is set to flagged and an AuditLog record is written with action=QUESTION_FLAGGED.
- Given the Question record is already flagged, when the flag request is received, then the response is 409 Conflict.
- Given a question is flagged, when the Module 2 Step Functions bank lookup GSI query executes, then the flagged question is not returned in the result set.
- Given the flag action succeeds, then an AuditLog record is written with actorId, actorRole=super_admin, action=QUESTION_FLAGGED, targetEntityId=questionId, ipAddress, userAgent, and timestamp.

**FR-A-032** | super_admin can unflag a QuestionBank entry | Must |
- Given an authenticated super_admin calls PATCH /admin/question-bank/{questionId}/unflag, when the Question record exists and has status=flagged, then the status field is set to active and an AuditLog record is written with action=QUESTION_UNFLAGGED.
- Given the Question record does not have status=flagged (it is active or deleted), when the unflag request is received, then the response is 409 Conflict.
- Given the unflag action succeeds, then an AuditLog record is written with actorId, actorRole=super_admin, action=QUESTION_UNFLAGGED, targetEntityId=questionId, ipAddress, userAgent, and timestamp.

**FR-A-033** | super_admin can soft-delete a QuestionBank entry | Must |
- Given an authenticated super_admin calls DELETE /admin/question-bank/{questionId}, when the Question record exists and has status=active or status=flagged, then the status field is set to deleted and an AuditLog record is written with action=QUESTION_SOFT_DELETED.
- Given a question has status=deleted, when any query runs against the QuestionBank table, then the question is not returned in any bank lookup result (Module 2 Step Functions GSI queries filter on status=active only).
- Given a question has status=deleted, when a super_admin calls GET /admin/question-bank?status=deleted, then the soft-deleted question is visible in the response (soft-deleted records are readable by super_admin for audit purposes).
- Given the soft-delete action succeeds, then an AuditLog record is written with actorId, actorRole=super_admin, action=QUESTION_SOFT_DELETED, targetEntityId=questionId, ipAddress, userAgent, and timestamp.
- Given a question has been soft-deleted and may be referenced in historical Worksheet records, the status=deleted record is never permanently removed from DynamoDB.

**FR-A-034** | super_admin can filter and list QuestionBank entries by status | Must |
- Given an authenticated super_admin calls GET /admin/question-bank with optional parameters status, grade, and subject, when the request is processed, then the response returns paginated Question records matching all supplied filters.
- Given no status filter is supplied, when the query is processed, then all questions regardless of status are returned (super_admin sees the full bank).
- Given a caller with any role other than super_admin calls any admin/question-bank endpoint, then the Lambda Authorizer returns 403.

### 5.5 AI Cost Dashboard

**FR-A-040** | Cost dashboard aggregates GenerationLog for token usage by time window | Must |
- Given an authenticated super_admin calls GET /admin/cost-dashboard?window=24h, GET /admin/cost-dashboard?window=7d, or GET /admin/cost-dashboard?window=30d, when the request is processed, then the response contains total tokens consumed in the specified window, cost estimate per model using pricing constants from the Config table (not hardcoded values), average tokens per worksheet broken down by subject and grade, and generation success rate versus failure rate versus retry rate.
- Given the Config table does not contain pricing constants for a model seen in GenerationLog, when the cost estimate is computed, then that model's cost is shown as unpriced rather than causing an error.
- Given a caller with any role other than super_admin calls any /admin/cost-dashboard endpoint, then the Lambda Authorizer returns 403.
- Given the cost dashboard query covers a 30-day window, when the query completes, then the response is returned within 2 seconds.

**FR-A-041** | Cost dashboard exposes top-10 most token-expensive generation requests | Must |
- Given an authenticated super_admin calls GET /admin/cost-dashboard/top-expensive, when the request is processed, then the response contains the ten GenerationLog records with the highest token count in the last 30 days, each record containing all GenerationLog fields.
- Given a super_admin clicks a record in the top-10 list in the admin console UI, when the click is processed, then the full GenerationLog record for that generation is displayed.

**FR-A-042** | Daily token budget ceiling is stored in Config table | Must |
- Given the Config table contains a record with PK=CONFIG#DAILY_TOKEN_BUDGET, when a super_admin calls GET /admin/config/DAILY_TOKEN_BUDGET, then the current ceiling value is returned.
- Given a super_admin calls PUT /admin/config/DAILY_TOKEN_BUDGET with a new integer value, when the type validation passes, then the Config record is updated and an AuditLog record is written with action=CONFIG_UPDATED.

**FR-A-043** | Module 2 worksheet generation Lambda enforces daily token budget ceiling | Must |
- Given the daily token budget ceiling is set in the Config table, when the worksheet generation Lambda is invoked via POST /worksheet, then the Lambda reads the current rolling 24-hour token consumption from GenerationLog before starting the Step Functions execution.
- Given the rolling 24-hour token consumption equals or exceeds the CONFIG#DAILY_TOKEN_BUDGET value, when the generation Lambda receives a POST /worksheet request, then it returns 503 Service Temporarily Unavailable with a Retry-After header (value: 3600 seconds) and does not start the Step Functions execution.
- Given the rolling 24-hour token consumption is below the budget ceiling, when the generation Lambda receives a POST /worksheet request, then the Step Functions execution proceeds normally.
- Given the Config table is unreachable when the budget check runs, when the Lambda catches the error, then the generation proceeds (fail-open) and the error is logged to CloudWatch. The budget check must never block worksheet generation due to a transient infrastructure failure.

### 5.6 Config Table Editor

**FR-A-050** | super_admin can read all Config table records | Must |
- Given an authenticated super_admin calls GET /admin/config, when the request is processed, then all Config records are returned, including CONFIG#SCHEMA, CONFIG#ALLOWED_MODELS, and all operational config keys.
- Given an authenticated super_admin calls GET /admin/config/{configType}, when the record exists, then the full Config record is returned including the key's defined type from CONFIG#SCHEMA.

**FR-A-051** | Config writes are type-validated before being applied | Must |
- Given an authenticated super_admin calls PUT /admin/config/{configType} with a value, when the request is processed, then the handler reads the type definition for that configType from CONFIG#SCHEMA before writing.
- Given the submitted value does not match the defined type (string, number, boolean, string-enum, or string-array), when the type check fails, then the request is rejected with 400 and a message naming the expected type.
- Given the configType is CONFIG#AI_MODEL, when the submitted value is not present in CONFIG#ALLOWED_MODELS, then the request is rejected with 400 regardless of type validity.
- Given all validations pass and the write succeeds, then an AuditLog record is written with action=CONFIG_UPDATED, targetEntityId=configType, beforeState containing the previous value, afterState containing the new value.

**FR-A-052** | Every Config write produces an AuditLog record | Must |
- Given any Config write completes successfully, then an AuditLog record is written with actorId, actorRole=super_admin, action=CONFIG_UPDATED, targetEntityId=configType, beforeState, afterState, ipAddress, userAgent, and timestamp.
- Given a Config write fails after the AuditLog write attempt, when the failed audit write is caught, then the error is logged to CloudWatch but the Config write result is not affected.

**FR-A-053** | Config writes do not require a Lambda redeploy | Must |
- Given a super_admin updates any Config value via PUT /admin/config/{configType}, when the change is written to DynamoDB, then the change takes effect within 60 seconds for all running Lambda functions without a Lambda redeploy or CDK stack update.

### 5.7 School and School Admin Management

**FR-A-060** | super_admin can create a school | Must |
- Given an authenticated super_admin calls POST /admin/schools with schoolName, gradeRange (minGrade, maxGrade), activeSubjects, and optional district and address, when the request is processed, then a School record is created with status=active, createdBy set to the super_admin's userId, and an AuditLog record is written with action=SCHOOL_CREATED.
- Given any required field (schoolName, minGrade, maxGrade) is absent, when the request is processed, then the response is 400 with a validation message.
- Given the school is created, when an AuditLog record is written, then it contains actorId, actorRole=super_admin, action=SCHOOL_CREATED, targetEntityId=schoolId, afterState containing the full School record, ipAddress, userAgent, and timestamp.

**FR-A-061** | super_admin can view and update a school record | Must |
- Given an authenticated super_admin calls GET /admin/schools, when the request is processed, then all School records are returned regardless of status.
- Given an authenticated super_admin calls GET /admin/schools/{schoolId}, when the school exists, then the full School record is returned including the list of schoolAdminIds.
- Given an authenticated super_admin calls PATCH /admin/schools/{schoolId} with updated fields, when the request is processed, then the School record is updated and an AuditLog record is written with action=SCHOOL_UPDATED, beforeState, and afterState.

**FR-A-062** | super_admin can assign a school_admin to a school | Must |
- Given an authenticated super_admin calls PATCH /admin/users/{userId}/role with role=school_admin and a schoolId in the request body, when the target user exists, then the User record is updated with role=school_admin, a SchoolUserLink record is created with PK=SCHOOL#schoolId, SK=USER#userId, role=school_admin, linkedAt, linkedBy=super_admin userId, status=active, and the schoolId is added to the School record's schoolAdminIds array.
- Given the schoolId supplied does not exist in the School table, when the assignment is requested, then the response is 400.
- Given the assignment succeeds, then an AuditLog record is written with action=ROLE_CHANGE and action=SCHOOL_ADMIN_ASSIGNED in the same request lifecycle.

**FR-A-063** | Class records gain a schoolId field | Must |
- Given the Module 7 data model migration runs, when a teacher affiliated with a school creates a Class record, then the Class record has its schoolId field set to the school the teacher is affiliated with (derived from the teacher's SchoolUserLink).
- Given a teacher is not affiliated with any school, when they create a Class record, then the schoolId field remains null.

### 5.8 Audit Log

**FR-A-070** | AuditLog is append-only at the application layer | Must |
- Given any state-changing admin action completes successfully, when the handler writes an AuditLog record, then it uses DynamoDB PutItem only (never UpdateItem or DeleteItem on the AuditLog table).
- Given any Lambda function or IAM role attempts to call UpdateItem or DeleteItem against the AuditLog table, then this is treated as a defect — no such call may exist in any codebase path.
- Given a super_admin calls any admin endpoint that is not a read-only GET, when the action succeeds, then an AuditLog record is written. If the AuditLog write fails, the error is logged to CloudWatch but the primary action result is not reversed.

**FR-A-071** | AuditLog records include all required fields | Must |
- Given an AuditLog record is written, when it is read back, then it contains: auditId (UUID v4), actorId, actorRole, action (from the defined enum), targetEntityType, targetEntityId, beforeState (JSON string or null), afterState (JSON string or null), ipAddress, userAgent, and timestamp (ISO-8601).

**FR-A-072** | super_admin can query AuditLog by actor or target | Must |
- Given an authenticated super_admin calls GET /admin/audit-log?actorId={userId}, when the query runs, then it uses GSI 1 (actorId + timestamp) and returns all AuditLog records for that actor in reverse-chronological order.
- Given an authenticated super_admin calls GET /admin/audit-log?targetEntityId={entityId}, when the query runs, then it uses GSI 2 (targetEntityId + timestamp) and returns all AuditLog records targeting that entity.
- Given an authenticated super_admin calls GET /admin/audit-log with from and to ISO-8601 timestamp parameters, when the query runs, then the results are filtered to the specified time range.
- Given a GSI query on AuditLog covers up to 10,000 records for a single actor, when the query completes, then the response is returned within 500 milliseconds.
- Given a caller with role=school_admin calls GET /admin/audit-log, when the query runs, then only AuditLog records where actorId matches the school_admin's own userId are returned (a school_admin may only see their own actions).

**FR-A-073** | AuditLog has no TTL and is retained indefinitely | Must |
- Given the AuditLog table is provisioned in CDK, when the table definition is inspected, then no TTL attribute is configured.
- Given a record is written to the AuditLog table, when any amount of time passes, then the record remains in the table until a super_admin explicitly configures a retention policy in the Config table. In Phase 2A no retention policy exists and all records are retained permanently.

### 5.9 Compliance Log

**FR-A-080** | ComplianceLog is written before any COPPA deletion step | Must |
- Given a COPPA deletion is initiated, when the handler begins, then a ComplianceLog record is written to DynamoDB as the very first operation.
- Given the ComplianceLog write fails, when the error is caught, then no deletion step is executed. The handler returns 500 with the message that the compliance record could not be persisted and the deletion was aborted.
- Given the ComplianceLog write succeeds, then the deletion steps proceed. The ordering constraint — ComplianceLog write must precede all deletions — is a hard requirement, not a recommendation.

**FR-A-081** | ComplianceLog records contain all required fields | Must |
- Given a ComplianceLog record is written at the start of a COPPA deletion, when it is read back, then it contains: requestId (UUID v4), requestType=coppa-deletion, requestedBy (super_admin userId), targetUserId, startedAt (ISO-8601), legalBasis, status (in-progress at creation), completedAt (null at creation), deletedEntities (null at creation), errorState (null at creation).
- Given the deletion completes successfully, when the ComplianceLog record is updated, then completedAt is set, deletedEntities lists each entity type deleted with its count, and status=completed.
- Given any deletion step fails, when the error is appended to the ComplianceLog, then errorState contains the step name, error message, and the count of records processed up to the point of failure.

**FR-A-082** | ComplianceLog has no TTL and is retained permanently | Must |
- Given the ComplianceLog table is provisioned in CDK, when the table definition is inspected, then no TTL attribute is configured.
- Given any ComplianceLog record is written, when any amount of time passes, then the record remains in the table. No admin console endpoint exposes a delete operation on ComplianceLog records.

**FR-A-083** | super_admin can view ComplianceLog | Must |
- Given an authenticated super_admin calls GET /admin/compliance-log, when the request is processed, then all ComplianceLog records are returned in reverse-chronological order by startedAt.
- Given a caller with any role other than super_admin calls GET /admin/compliance-log, then the Lambda Authorizer returns 403.

## 7. Non-Functional Requirements

### 7.1 Security

**NFR-SEC-001** | The admin console CloudFront distribution for super_admin access must be IP-restricted to the Learnfyra ops team's office and VPN CIDR ranges. This restriction is applied at the CloudFront level (WAF IP set or CloudFront geo and IP restriction), not at the application layer. School admin access to the same distribution is not IP-restricted but requires a valid JWT with role=school_admin.

**NFR-SEC-002** | The AuditLog must be enforced as append-only at the application layer. No handler in any Lambda function may call DynamoDB UpdateItem or DeleteItem against the AuditLog table. This must be verifiable by code review and must be included in the PR checklist for any handler that touches AuditLog.

**NFR-SEC-003** | The COPPA deletion endpoint and the force-logout endpoint must each require explicit UI-level double confirmation before the API call is made. For COPPA deletion, the confirmation requires typing the user's email. For force-logout, the confirmation requires an acknowledgement dialog.

**NFR-SEC-004** | No PII must appear in CloudWatch logs at any log level. User email addresses, display names, and Cognito sub identifiers must be redacted or omitted from all Lambda log statements. ComplianceLog records must be stored in DynamoDB only, not echoed to CloudWatch.

**NFR-SEC-005** | All /admin/* and /school/* routes must be protected by the Lambda Authorizer. The Authorizer must validate the JWT signature, check expiry, confirm the role claim, and check the suspended flag on every request.

### 7.2 Performance

**NFR-PERF-001** | AuditLog GSI queries must return within 500 milliseconds for up to 10,000 audit records per actor or per target entity.

**NFR-PERF-002** | Cost dashboard queries must return within 2 seconds for the 30-day window.

**NFR-PERF-003** | The Lambda Authorizer suspended flag check adds at most one DynamoDB GetItem call per authenticated request when the in-memory cache has expired. With a 5-minute cache TTL, this adds one DynamoDB read per userId every 5 minutes at most, which is acceptable at projected Phase 2 admin user volumes.

**NFR-PERF-004** | User search (GET /admin/users) must return within 2 seconds for a dataset of up to 10,000 users.

### 7.3 Data Integrity

**NFR-DI-001** | ComplianceLog write must complete before any deletion step in the COPPA flow begins. This is a hard ordering constraint enforced in the Lambda handler, not an eventually consistent pattern.

**NFR-DI-002** | Partial deletion state must be recorded in the ComplianceLog errorState field if any COPPA deletion step fails. The record must accurately reflect which steps completed, which failed, and the counts of records deleted at each step.

**NFR-DI-003** | AuditLog writes are fire-and-forget from the perspective of the primary action. A failed AuditLog write must not roll back the primary action. The error must be logged to CloudWatch with sufficient context to reconstruct the audit record manually if needed.

### 7.4 Compliance

**NFR-COMP-001** | COPPA deletion must remove all entity types specified in FR-A-023 for the target user. The ComplianceLog must record the count of each entity type deleted.

**NFR-COMP-002** | ComplianceLog retention is permanent. No TTL attribute is set on the ComplianceLog table and no admin console endpoint exposes a delete operation on ComplianceLog records.

**NFR-COMP-003** | The ops team's legal counsel must review the COPPA deletion flow implementation before Phase 2A is released to production. This is an external dependency and must be scheduled before the Phase 2A go/no-go gate.

### 7.5 Isolation

**NFR-ISO-001** | A school_admin must not be able to query any data outside their schoolId scope. All /school/* Lambda handlers must extract the caller's schoolId from their SchoolUserLink record (not from a request parameter) and use it as the exclusive scope for all DynamoDB queries.

**NFR-ISO-002** | Enumeration of school IDs by a school_admin must return 403 (not 404) for unaffiliated schools. This prevents an attacker with a valid school_admin JWT from discovering which school IDs exist.

### 7.6 Auditability

**NFR-AUD-001** | Every state-changing action by any admin role (super_admin or school_admin) must produce an AuditLog record. Read-only GET requests do not require an AuditLog record. This applies to all endpoints listed in Section 9.

## 9. API Endpoint Definitions

All endpoints under /admin/* require role=super_admin in the JWT unless otherwise noted. All endpoints under /school/* require role=school_admin in the JWT. The Lambda Authorizer enforces these rules before the handler executes.

---

### 9.1 Platform Admin Endpoints

**GET /admin/users**

- Auth: role=super_admin required. Lambda Authorizer returns 403 for any other role.
- Query parameters: search (string, optional), role (string enum, optional), suspended (boolean, optional), page (integer, default 1), pageSize (integer, default 50, max 100).
- Response 200: { users: [ { userId, email, displayName, role, suspended, createdAt, lastLoginAt } ], totalCount, page, pageSize }
- Response 400: { error: "validation message" } — invalid parameter types.
- Response 403: { error: "Forbidden" } — caller is not super_admin.
- RBAC rule: Lambda Authorizer must confirm role=super_admin before routing to handler.

**GET /admin/users/{userId}**

- Auth: role=super_admin required.
- Path parameter: userId (string, UUID v4).
- Response 200: { userId, email, displayName, role, suspended, createdAt, lastLoginAt, generationCount, attemptCount }
- Response 403: caller not super_admin.
- Response 404: userId not found in Users table.
- RBAC rule: Lambda Authorizer must confirm role=super_admin.

**PATCH /admin/users/{userId}/suspend**

- Auth: role=super_admin required.
- Path parameter: userId (string, UUID v4).
- Request body: { reason: string (optional, for AuditLog afterState) }
- Response 200: { userId, suspended: true }
- Response 403: caller not super_admin.
- Response 404: userId not found.
- Response 409: user already suspended.
- AuditLog requirement: On success, an AuditLog record must be written with action=USER_SUSPENDED, targetEntityId=userId, actorId, actorRole=super_admin, beforeState={suspended:false}, afterState={suspended:true}, ipAddress, userAgent, timestamp.

**PATCH /admin/users/{userId}/unsuspend**

- Auth: role=super_admin required.
- Path parameter: userId (string, UUID v4).
- Response 200: { userId, suspended: false }
- Response 403: caller not super_admin.
- Response 404: userId not found.
- Response 409: user is not currently suspended.
- AuditLog requirement: On success, an AuditLog record must be written with action=USER_UNSUSPENDED, targetEntityId=userId, actorId, actorRole=super_admin, beforeState={suspended:true}, afterState={suspended:false}, ipAddress, userAgent, timestamp.

**POST /admin/users/{userId}/force-logout**

- Auth: role=super_admin required.
- Path parameter: userId (string, UUID v4).
- Request body: { confirmed: true } — UI must set this only after the confirmation dialog is accepted.
- Response 200: { userId, loggedOut: true }
- Response 400: confirmed not true in body.
- Response 403: caller not super_admin.
- Response 404: userId not found.
- Response 502: Cognito AdminUserGlobalSignOut call failed.
- AuditLog requirement: On success, an AuditLog record must be written with action=FORCE_LOGOUT, targetEntityId=userId, actorId, actorRole=super_admin, ipAddress, userAgent, timestamp.

**PATCH /admin/users/{userId}/role**

- Auth: role=super_admin required.
- Path parameter: userId (string, UUID v4).
- Request body: { role: string, schoolId: string (required when role=school_admin, ignored otherwise) }
- Allowed role values: student, teacher, parent, school_admin, super_admin.
- Response 200: { userId, previousRole, newRole }
- Response 400: invalid role value, or role=school_admin without schoolId, or schoolId does not exist.
- Response 403: caller not super_admin, or caller is attempting to change their own role.
- Response 404: userId not found.
- AuditLog requirement: On success, an AuditLog record must be written with action=ROLE_CHANGE, targetEntityId=userId, actorId, actorRole=super_admin, beforeState={role:previousRole}, afterState={role:newRole}, ipAddress, userAgent, timestamp.

**DELETE /admin/users/{userId}**

- Auth: role=super_admin required.
- Path parameter: userId (string, UUID v4).
- Request body: { confirmationToken: string } — token issued after UI double-confirmation.
- Response 200: { requestId, status: "completed", deletedEntities: [ { entityType, count } ] }
- Response 400: confirmationToken absent or invalid.
- Response 403: caller not super_admin.
- Response 404: userId not found.
- Response 500: ComplianceLog write failed (deletion aborted) or a deletion step failed (partial deletion).
- ComplianceLog requirement: A ComplianceLog record must be written and persisted to DynamoDB as the first operation in this handler, before any deletion step begins. If the ComplianceLog write fails, the handler must return 500 immediately without executing any deletion step.
- AuditLog requirement: On completion (success or partial failure), an AuditLog record must be written with action=COPPA_DELETION, targetEntityId=userId, actorId, actorRole=super_admin, ipAddress, userAgent, timestamp.

**GET /admin/question-bank**

- Auth: role=super_admin required.
- Query parameters: status (string enum: active, flagged, deleted; optional), grade (integer 1–10; optional), subject (string enum; optional), page, pageSize.
- Response 200: { questions: [ { questionId, status, grade, subject, type, question, createdAt } ], totalCount, page, pageSize }
- Response 403: caller not super_admin.
- RBAC rule: Lambda Authorizer must confirm role=super_admin.

**PATCH /admin/question-bank/{questionId}/flag**

- Auth: role=super_admin required.
- Path parameter: questionId (string).
- Response 200: { questionId, status: "flagged" }
- Response 403: caller not super_admin.
- Response 404: questionId not found.
- Response 409: question already flagged or already deleted.
- AuditLog requirement: On success, an AuditLog record must be written with action=QUESTION_FLAGGED, targetEntityId=questionId, actorId, actorRole=super_admin, beforeState={status:active}, afterState={status:flagged}, ipAddress, userAgent, timestamp.

**PATCH /admin/question-bank/{questionId}/unflag**

- Auth: role=super_admin required.
- Path parameter: questionId (string).
- Response 200: { questionId, status: "active" }
- Response 403: caller not super_admin.
- Response 404: questionId not found.
- Response 409: question is not currently flagged.
- AuditLog requirement: On success, an AuditLog record must be written with action=QUESTION_UNFLAGGED, targetEntityId=questionId, actorId, actorRole=super_admin, beforeState={status:flagged}, afterState={status:active}, ipAddress, userAgent, timestamp.

**DELETE /admin/question-bank/{questionId}**

- Auth: role=super_admin required.
- Path parameter: questionId (string).
- Response 200: { questionId, status: "deleted" }
- Response 403: caller not super_admin.

## 10. User Journey Flows

### 10.1 Super Admin Bootstrapped at CDK Deployment — First Login

1. The Learnfyra DevOps engineer passes the designated ops team member's Cognito-registered email address as a CDK context parameter at deployment time.
2. The CDK bootstrap Lambda or CDK custom resource checks whether the Cognito user for that email exists. If the user does not exist in Cognito, the deployment fails with a clear error.
3. If the Cognito user exists, the bootstrap process writes a User record in DynamoDB with role=super_admin, userId=cognitoSub, email, and createdAt.
4. The bootstrap writes a ComplianceLog record with requestType=bootstrap, targetUserId=cognitoSub, and legalBasis=initial-deployment to establish a permanent record of the first privilege grant.
5. The first super_admin navigates to admin.learnfyra.com (or the environment equivalent) from an IP within the approved ops CIDR range.
6. They sign in using their existing Cognito credentials (Google OAuth or email/password).
7. The Lambda Authorizer validates the JWT, confirms role=super_admin, and checks the IP against the CloudFront IP restriction. Access is granted.
8. The admin console home page loads, showing the platform overview dashboard.

### 10.2 Super Admin Creates a School and Assigns a School Admin

1. The super_admin navigates to the Schools section of the admin console.
2. They click "Create School" and fill in: school name, grade range (minGrade and maxGrade), active subjects, and optionally district and address.
3. On submission, the UI sends POST /admin/schools. The API creates a School record and returns the new schoolId.
4. An AuditLog record is written with action=SCHOOL_CREATED.
5. The super_admin navigates to User Management and searches for the user who will become the school admin (by email or name).
6. They open that user's profile and click "Assign School Admin Role", selecting the newly created school from a dropdown.
7. On confirmation, the UI sends PATCH /admin/users/{userId}/role with role=school_admin and schoolId.
8. The API updates the User record to role=school_admin, creates a SchoolUserLink record, and adds the userId to the School record's schoolAdminIds.
9. An AuditLog record is written with action=ROLE_CHANGE.
10. The now-school_admin user receives no automatic notification in Phase 2A (notification is a Phase 3 feature). The super_admin informs them through out-of-band means.

### 10.3 Super Admin Suspends a User and Force-Logs Them Out

1. The super_admin searches for the target user in User Management.
2. They open the user's profile and click "Suspend Account".
3. The UI presents a confirmation dialog: "Suspending this account will prevent the user from making any authenticated requests within 5 minutes. Proceed?"
4. On confirmation, the UI sends PATCH /admin/users/{userId}/suspend.
5. The API sets suspended=true on the User record and writes an AuditLog record.
6. The super_admin sees the user's status updated to Suspended in the UI.
7. For security-critical cases, the super_admin clicks "Force Immediate Logout" on the same profile page.
8. A second confirmation dialog appears: "This will immediately invalidate all active sessions and refresh tokens for this user. This action cannot be undone. Proceed?"
9. On confirmation, the UI sends POST /admin/users/{userId}/force-logout with { confirmed: true }.
10. The API calls Cognito AdminUserGlobalSignOut for the user's Cognito username.
11. On success, an AuditLog record is written with action=FORCE_LOGOUT and the user's active sessions are terminated immediately.

### 10.4 Super Admin Processes a COPPA Deletion Request End-to-End

1. The super_admin receives a COPPA deletion request through the appropriate legal intake channel (out-of-band — not tracked in Learnfyra itself in Phase 2A).
2. They navigate to User Management in the admin console and search for the target user by email.
3. They open the user's profile and click "COPPA Deletion".
4. The UI presents the double-confirmation modal: "To permanently delete all data for this user, type their email address: [___________]".
5. The super_admin types the user's exact email address. If the typed value does not match, the confirm button remains disabled.
6. On match, the confirm button becomes active. The super_admin clicks "Permanently Delete All Data".
7. A short-lived confirmation token is generated client-side and the UI sends DELETE /admin/users/{userId} with the token.
8. The handler immediately writes a ComplianceLog record with status=in-progress and legalBasis populated.
9. The handler executes the deletion steps in order: WorksheetAttempt, UserProgress, ParentChildLink, StudentAssignmentStatus, Class roster removal, User record, Cognito account.
10. On completion, the ComplianceLog record is updated with status=completed, completedAt, and deletedEntities.
11. An AuditLog record is written with action=COPPA_DELETION.
12. The UI displays a summary showing each entity type and the count of records deleted.
13. If any step fails, the UI displays the partial deletion state and the error, and the super_admin is instructed to contact engineering for manual resolution.

### 10.5 Super Admin Flags and Soft-Deletes a Low-Quality QuestionBank Entry

1. The super_admin navigates to Question Bank Moderation in the admin console.
2. They filter by subject and grade to narrow the list, and identify a question with a factual error.
3. They click "Flag" on the question. The UI sends PATCH /admin/question-bank/{questionId}/flag.
4. The question's status is set to flagged and an AuditLog record is written.
5. The question immediately stops appearing in Module 2 Step Functions bank lookup queries.
6. The super_admin reviews the flagged question in context and confirms it should be removed permanently.
7. They click "Soft Delete" on the flagged question. The UI sends DELETE /admin/question-bank/{questionId}.
8. The question's status is set to deleted and an AuditLog record is written.
9. The question remains visible in the admin console when the super_admin filters by status=deleted, but is excluded from all production bank queries.

### 10.6 Super Admin Raises the Daily Token Budget Ceiling

1. The super_admin navigates to the Config Table Editor in the admin console.
2. They see the current CONFIG#DAILY_TOKEN_BUDGET value.
3. They note from the AI cost dashboard that the current value is causing legitimate generation requests to be rejected with 503 during peak usage.
4. They click "Edit" on the CONFIG#DAILY_TOKEN_BUDGET record and enter the new integer value.
5. The UI sends PUT /admin/config/DAILY_TOKEN_BUDGET with the new value.
6. The API reads the type definition for this key from CONFIG#SCHEMA (type: number), validates that the submitted value is a number, and writes the update.
7. An AuditLog record is written with action=CONFIG_UPDATED, beforeState, and afterState.
8. Within 60 seconds, the worksheet generation Lambda reads the updated value on its next Config cache refresh and begins applying the new ceiling.

### 10.7 School Admin Invites a Teacher to Their School

1. The school_admin navigates to the Teachers section of the school admin console.
2. They click "Invite Teacher".
3. The UI sends POST /school/teachers/invite.
4. An invite code is generated and returned. The school_admin copies the code and shares it with the teacher via email or another channel.
5. An AuditLog record is written with action=TEACHER_INVITED.
6. The teacher receives the code, signs in or registers on Learnfyra, and enters the invite code in their account settings.
7. The system validates the invite code, creates a SchoolUserLink record for the teacher, and returns a success message to the teacher.
8. The school_admin refreshes the Teachers list and sees the new teacher in the roster with status=active.

### 10.8 School Admin Runs a Bulk Worksheet Assignment Across Three Classes

1. The school_admin navigates to the Bulk Assignment section.
2. They search for a worksheet by title, grade, or subject and select one.
3. They select three classes from a multi-select list. All three classes belong to their school.
4. They set the due date and allowed attempts.
5. The UI sends POST /school/bulk-assign with worksheetId, three classIds, dueDate, and allowedAttempts.
6. The handler validates that all three classIds are affiliated with the school_admin's schoolId.
7. Three parallel POST /assignments calls are issued, one per class.
8. The response returns a summary: { results: [ {classId1, status: "assigned"}, {classId2, status: "assigned"}, {classId3, status: "assigned"} ] }.
9. An AuditLog record is written with action=BULK_ASSIGNMENT_CREATED, listing all three classIds and the worksheetId.
10. The school_admin sees the confirmation summary in the UI.

### 10.9 School Admin Reviews Analytics and Identifies a Weak Grade/Subject Combination

1. The school_admin navigates to the School Analytics Dashboard.
2. The UI calls GET /school/analytics.
3. The response returns subject-level accuracy, grade-level accuracy, and teacher completion rates for all classes in the school.
4. The dashboard renders a grid of subjects by grade. The school_admin sees that Grade 3 Science has an average accuracy of 52%, which is below the threshold of 60%.
5. The Grade 3 Science cell is visually highlighted in the UI.
6. The school_admin contacts the Grade 3 teacher to discuss additional practice materials.

## 11. Open Questions

**OQ-7-001** | Teacher invite code mechanism

- Question: Should the school teacher invite code use the same ParentInviteCode table as the parent linking flow (Module 5), or a new SchoolTeacherInvite table?
- Impact if unresolved: The data model for POST /school/teachers/invite cannot be finalised and the task cannot be allocated to DEV.
- Suggested default: Create a new SchoolTeacherInvite table (PK: INVITE#<code>, SK: METADATA, attributes: schoolId, invitedBy, createdAt, expiresAt, status) rather than reusing ParentInviteCode. The two use cases have different expiry semantics and the tables would share a PK prefix pattern but different attributes. Separation is cleaner.

**OQ-7-002** | AuditLog retention policy configuration

- Question: The design states that a super_admin may configure a retention policy in Config in the future. Should Phase 2A include the Config key (CONFIG#AUDIT_RETENTION_DAYS) in CONFIG#SCHEMA as a defined but unset key, or should it be entirely absent until Phase 3?
- Impact if unresolved: The Config editor in Phase 2A may confuse admins if the key is present but has no effect.
- Suggested default: Define CONFIG#AUDIT_RETENTION_DAYS in CONFIG#SCHEMA as a number type with a default of null (meaning indefinite). The AuditLog Lambda checks for this value but takes no action if it is null. This makes the Phase 3 feature a one-line change.

**OQ-7-003** | School admin console access for school_admin role

- Question: The admin console at admin.learnfyra.com is IP-restricted for super_admin access. Is school_admin access to the same CloudFront distribution not IP-restricted, or does the school_admin access a different URL on a different distribution?
- Impact if unresolved: The CDK stack for the admin console cannot be finalised without knowing whether there is one distribution or two.
- Suggested default: One distribution (admin.learnfyra.com) with WAF rules that allow all IPs to reach /school/* paths, but restrict /admin/* paths to approved ops CIDRs. The Authorizer already enforces role=super_admin on /admin/* and role=school_admin on /school/*, so the WAF restriction is defence-in-depth for super_admin paths only.

**OQ-7-004** | Partial COPPA deletion notification

- Question: When a COPPA deletion step fails and leaves a partial deletion state, how is the super_admin notified? The design says the error is presented in the UI response, but what if the super_admin has closed the browser by the time the failure occurs?
- Impact if unresolved: Legal risk. A partial COPPA deletion with no notification to the super_admin may leave the platform in violation of COPPA if the failure is not discovered.
- Suggested default: In Phase 2A, the handler response always includes the full partial deletion state. If the error occurs asynchronously (future async flow), a CloudWatch alarm on ComplianceLog errorState writes should be configured. For Phase 2A (synchronous Lambda), the super_admin will see the error in the UI response. A Phase 3 email notification for partial deletion failures is recommended.

**OQ-7-005** | Bootstrap ComplianceLog record

- Question: Section 10.1 describes writing a ComplianceLog record at bootstrap. This is not a COPPA deletion. Should the requestType enum be extended to include bootstrap at the data model level?
- Impact if unresolved: The data model for ComplianceLog must be updated before implementation if bootstrap is in scope.
- Suggested default: Extend requestType to include bootstrap as a valid value. The ComplianceLog is the appropriate place to record the initial privilege grant — it is a compliance event, not an audit event, because it is not triggered by a user action and predates the AuditLog.

## 12. Dependencies

### Internal Module Dependencies

All of Modules 1 through 6 must be complete and stable before Module 7 implementation begins. Specific dependencies:

- Module 1 (Auth and Identity): Lambda Authorizer must be operational before the suspended flag amendment (FR-A-013) and the new role values (super_admin, school_admin) can be added. The authorizer amendment is a modification to the existing authorizer Lambda, not a new function.
- Module 2 (Worksheet Generator): The QuestionBank table must be live and the Step Functions bank lookup GSI must be operational before the status field amendment (FR-A-030) and the daily token budget ceiling check (FR-A-043) can be applied.
- Module 5 (Teacher and Parent): SchoolUserLink data model design informed by the ParentChildLink and ParentInviteCode patterns. The Class table must be live before the schoolId amendment (FR-A-063) can be applied.
- Module 6 (Infrastructure and CDK): The multi-environment CDK stack must support a separate CloudFront distribution with WAF IP restriction before the admin console can be deployed.

### External Dependencies

- Cognito Admin API access: The force-logout feature (POST /admin/users/{userId}/force-logout) requires the Lambda handler's IAM role to have permission to call Cognito AdminUserGlobalSignOut. This permission must be granted in CDK.
- CloudFront IP restriction configuration: The ops team must provide a list of approved IP CIDRs (office and VPN) before the admin console CDK stack can be deployed with IP restriction enabled. This list is a CDK parameter, not a hardcoded value.
- AWS Bedrock pricing constants: The AI cost dashboard requires pricing data per model per token. In Phase 2A, these values are stored in the Config table (not hardcoded and not pulled from the AWS Pricing API). The initial values must be populated in the Config table at deployment time via a CDK seed script.
- Legal review: The COPPA deletion flow must be reviewed by the ops team's legal counsel before Phase 2A is released to production (NFR-COMP-003).
