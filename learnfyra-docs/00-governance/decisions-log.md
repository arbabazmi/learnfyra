# Architecture & Governance Decisions Log

## ADR-001: JSON over DynamoDB for Worksheet Storage
**Date:** 2026-03
**Status:** Accepted

**Decision:** Store worksheet content (questions, answers, metadata) as JSON files in S3 (`solve-data.json`, `metadata.json`) rather than in DynamoDB.

**Rationale:**
- Write-once-read-many access pattern — no cross-worksheet querying needed
- Small per-worksheet payload (< 50KB typical)
- No extra AWS service cost beyond S3 already used for PDF/DOCX/HTML
- Native to Node.js, works with S3 lifecycle rules (7-day auto-expiry)
- DynamoDB reserved for relational data: users, attempts, classes, question bank

**Consequences:** All worksheet content retrieval goes through S3. Worksheet IDs must map to deterministic S3 key prefixes.

## ADR-002: Local-First Development
**Date:** 2026-03
**Status:** Accepted

**Decision:** All features must work on localhost:3000 before AWS deployment. Lambda handlers are called directly from Express routes without modification.

**Rationale:**
- Eliminates cold-start and deploy-cycle friction during active development
- Reduces AWS cost during development phase
- Forces clean handler abstraction (no Lambda-specific code paths)
- Local integration tests run without AWS credentials

**Consequences:** APP_RUNTIME env var controls which adapters are used. Every handler must be testable with a plain JavaScript function call using a mock event object.

## ADR-003: Bank-First Worksheet Assembly
**Date:** 2026-03
**Status:** Accepted

**Decision:** POST /api/generate queries the Question Bank before calling Claude AI. AI is called only for gap-fill when the bank has insufficient questions.

**Rationale:**
- Reduces Claude API cost and latency for repeat topic combinations
- Enforces question deduplication via SHA256 dedupeHash
- Enables per-student repeat caps without AI involvement
- generationMode field (`bank-only` | `ai-only` | `mixed`) makes provenance transparent

**Consequences:** M02 (Question Bank) must be built and stable before M03 (Worksheet Generator) can be completed. Questions generated by AI must be written back to the bank after validation.

## ADR-004: Cognito + Google OAuth for Authentication
**Date:** 2026-03
**Status:** Accepted

**Decision:** Use AWS Cognito Hosted UI with PKCE for Google OAuth. Local email/password accounts also backed by Cognito User Pool.

**Rationale:**
- Managed token issuance, rotation, and blacklisting reduces security implementation burden
- Google OAuth covers the majority of target user (teacher/student/parent) sign-in behavior
- Cognito integrates natively with Lambda Authorizer pattern on API Gateway
- No secrets stored client-side (PKCE eliminates client secret requirement)

**Consequences:** Each environment (dev/staging/prod) needs a separate Cognito User Pool and Google OAuth client ID. IaC agent owns Cognito CDK constructs.

## ADR-005: ARM_64 Lambda Architecture
**Date:** 2026-03
**Status:** Accepted

**Decision:** All Lambda functions use ARM_64 (Graviton2) architecture.

**Rationale:** 20% cheaper per GB-second than x86_64. Graviton2 is generally faster for Node.js workloads. All esbuild bundles are architecture-agnostic.

**Consequences:** CDK `NodejsFunction` construct must specify `architecture: lambda.Architecture.ARM_64` on every function.

## ADR-006: DynamoDB QB Adapter Pattern
**Date:** 2026-03
**Status:** Accepted

**Decision:** Question Bank access is abstracted behind a QB_ADAPTER interface. `QB_ADAPTER=local` uses JSON files; `QB_ADAPTER=dynamodb` uses AWS SDK v3.

**Rationale:**
- Enables full local development and unit testing without DynamoDB
- Same handler code runs in both environments without branching
- Follows the service adapter strategy established for all AWS dependencies

**Adapter Interface:**
```javascript
// getQuestions(grade, subject, topic, type, difficulty, count) → Question[]
// saveQuestion(question) → { dedupeHash, questionId }
// questionExists(dedupeHash) → boolean
```

## ADR-010: Frontend API Base URL and Config Injection Strategy
**Question:** OQ-1 (M01 FE)
**Decision:** `auth.js` reads `window.LEARNFYRA_CONFIG.apiBaseUrl` from a generated `config.js` file included in every HTML `<head>`. Locally, a committed `frontend/config.local.js` serves as the static equivalent. CI/CD generates `frontend/config.js` from env vars before `aws s3 sync`. No CloudFront Function.
**Rationale:** A generated `config.js` works identically in local Express dev and CloudFront prod. CloudFront Functions add cold-start latency and cannot be tested locally. A committed `config.local.js` keeps local dev zero-config.
**Local implementation:** `frontend/config.local.js` is committed and sets `window.LEARNFYRA_CONFIG = { apiBaseUrl: "http://localhost:3000", cognitoAuthorizeUrl: "...", clientId: "...", redirectUri: "http://localhost:3000/login.html" }`. Express `server.js` serves `frontend/` as static, so it is picked up automatically.
**AWS implementation:** CI/CD deploy step writes `frontend/config.js` from GitHub Actions secrets (`CLOUDFRONT_DOMAIN`, `COGNITO_AUTHORIZE_URL`, `COGNITO_CLIENT_ID`) before `aws s3 sync`. Every HTML page includes `<script src="/config.js"></script>`.
**Impact on specs:** DevOps must add `config.js` generation step to all three deploy workflows. `07-requirements/auth/frontend.md` OQ-3 is closed. `window.LEARNFYRA_CONFIG` shape: `{ apiBaseUrl, cognitoAuthorizeUrl, clientId, redirectUri }`.

## ADR-011: dedupeHash-index GSI on LearnfyraQuestionBank
**Question:** OQ-2 (M02)
**Decision:** Add `dedupeHash-index` GSI to `LearnfyraQuestionBank-{env}`. PK = `dedupeHash` (String). No sort key. Projection = `KEYS_ONLY`.
**Rationale:** Deduplication requires a point lookup by hash. The existing GSI-1 (lookupKey + typeDifficulty) cannot serve this. `KEYS_ONLY` minimises storage cost — stores only `dedupeHash` + `questionId`.
**Local implementation:** `QB_ADAPTER=local` iterates the local JSON file for dedupe checks. No GSI infrastructure needed locally.
**AWS implementation:** CDK `database.ts` adds GSI: `{ indexName: "dedupeHash-index", partitionKey: { name: "dedupeHash", type: AttributeType.STRING }, projectionType: ProjectionType.KEYS_ONLY }`.
**Impact on specs:** `03-data-design/dynamodb-design.md` must add `dedupeHash-index` to QuestionBank GSI table. IaC agent must update `infra/lib/constructs/database.ts`.

## ADR-012: matching Question JSON Schema and short-answer Keyword Delimiter
**Question:** OQ-3 (M04)
**Decision:** (a) `matching` questions store pairs in a `pairs` attribute (JSON array): `[{ "left": "term", "right": "definition" }]`. The `answer` field is set to sentinel string `"pairs"`. `options` field is absent. (b) `short-answer` keywords are pipe-delimited in the `answer` field: `"osmosis|membrane|diffusion"`.
**Rationale:** Structured `pairs` array preserves left/right relationship for dropdown rendering and per-pair scoring. Pipe-delimited answer keeps DynamoDB attribute type consistent (String) across all question types. Pipe is safe — unlikely in educational keywords.
**Local implementation:** `scorer.js` reads `question.pairs` for matching, `question.answer.split('|')` for short-answer. Both are pure JS, no env differences.
**AWS implementation:** No infrastructure change. `promptBuilder.js` must instruct Claude to output `pairs` as JSON array and short-answer answers as pipe-delimited strings.
**Impact on specs:** `03-data-design/dynamodb-design.md` must add `pairs` attribute to QuestionBank schema. `04-api-contracts/solving-api.md` must document `pairs` field in GET /api/solve response. `CLAUDE.md` canonical schema must be updated with both rules.

## ADR-013: aggregator.js Concurrent Write Safety and certificateBuilder.js Puppeteer Strategy
**Question:** OQ-4 (M05)
**Decision:** (a) `aggregator.js` uses DynamoDB `ADD` for `totalAttempts` and the incremental mean formula with `ConditionExpression` + `withRetry` for `avgScore`: `newAvg = oldAvg + (newScore - oldAvg) / newTotal`. (b) Local: Puppeteer bundled Chromium directly. Lambda: `@sparticuz/chromium` ARM64 layer (under 50MB compressed).
**Rationale:** Conditional expression prevents race condition from concurrent submits corrupting `avgScore`. Incremental mean is O(1) — no full history read on each submit. `@sparticuz/chromium` is the standard solution for Chromium on Lambda ARM64 within the 50MB layer limit.
**Local implementation:** `certificateBuilder.js` imports Puppeteer normally. Aggregator conditional write runs identically against `dynamodb-local` on `http://localhost:8000`. `DYNAMODB_ENDPOINT=http://localhost:8000` env var switches DynamoDB client endpoint.
**AWS implementation:** CDK `compute.ts` adds `@sparticuz/chromium` LayerVersion to submit and certificate Lambdas. `certificateBuilder.js` switches import based on `NODE_ENV`. `.env.example` must document `DYNAMODB_ENDPOINT`.
**Impact on specs:** `02-modules/reporting.md` must document incremental mean formula and retry strategy. IaC must add Chromium layer to CDK. DevOps must add `DYNAMODB_ENDPOINT` to `.env.example`.

## ADR-014: joinCode-index GSI on LearnfyraClasses
**Question:** OQ-5 (M06)
**Decision:** Add `joinCode-index` GSI to `LearnfyraClasses-{env}`. PK = `joinCode` (String). No sort key. Projection = `KEYS_ONLY`.
**Rationale:** Students join by entering a 6-character code. The only access pattern is "given joinCode, find classId." `KEYS_ONLY` stores only `joinCode` + `classId` — minimum cost. GSI belongs on `LearnfyraClasses` (not Memberships) because `joinCode` is a class attribute.
**Local implementation:** `CLASSES_ADAPTER=local` iterates local JSON file for join code lookup. `dynamodb-local` handles the GSI transparently for integration tests.
**AWS implementation:** CDK `database.ts` adds GSI: `{ indexName: "joinCode-index", partitionKey: { name: "joinCode", type: AttributeType.STRING }, projectionType: ProjectionType.KEYS_ONLY }`.
**Impact on specs:** `03-data-design/dynamodb-design.md` must add `joinCode-index` to LearnfyraClasses. IaC must update `infra/lib/constructs/database.ts`.
